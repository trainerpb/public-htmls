Brute-Force Methodologies

The Strategy: The Brute-Force paradigm is the most fundamental approach to algorithmic design. It systematically enumerates all possible candidates for the solution and checks whether each candidate satisfies the problem's statement. While it is mathematically guaranteed to find the correct solution (or definitively prove none exists), it makes absolutely no attempt to optimize. Because it leaves no stone unturned, it often scales terribly, leading to inefficient $O(n^2)$, $O(2^n)$, or $O(n!)$ time complexities.
Use Case Breakdown: Linear Search
Time: $O(n)$ Space: $O(1)$

What it says: Linear Search is the purest form of Brute Force. When tasked with finding a specific item in an array, it starts at index 0 and checks every single element sequentially.

Why it is Brute Force: It makes zero assumptions about the data. It doesn't check if the array is sorted (which would allow Binary Search). It just blindly plows forward. If the array has 1 billion elements and the target is at the very end, it will perform 1 billion comparisons.

Java: Linear Search
public int linearSearch(int[] arr, int target) {
    for (int i = 0; i < arr.length; i++) {
        if (arr[i] == target) return i;
    }
    return -1;
}

Use Case Breakdown: Naive String Match
Time: $O(n \times m)$ Space: $O(1)$

What it says: To find a short word (the pattern) inside a long sentence (the text), it slides a window over the text one character at a time.

Why it is Brute Force: If the pattern is "ABC" and the text is "ABXABC", it successfully matches 'A' and 'B', but fails on 'X'. A smart algorithm (like KMP) would realize it doesn't need to re-check the 'B'. But the Naive Brute Force algorithm immediately abandons all progress, shifts the pattern by exactly one space, and blindly starts comparing from scratch.

Java: Naive String Match
public void search(String txt, String pat) {
    int n = txt.length(), m = pat.length();
    for (int i = 0; i <= n - m; i++) {
        int j;
        for (j = 0; j < m; j++) {
            if (txt.charAt(i + j) != pat.charAt(j)) {
                break; // Mismatch! Reset j completely.
            }
        }
        if (j == m) System.out.println("Match at " + i);
    }
}

Use Case Breakdown: Travelling Salesperson Problem (TSP)
Time: $O(n!)$ Space: $O(n)$

What it says: A salesperson needs to visit a set of cities and return home via the absolute shortest route. The brute force solution calculates the total distance for every single possible permutation of city visits.

Why it is Brute Force: Because it calculates $n!$ routes. If you have 5 cities, there are 120 routes. If you have 15 cities, there are 1.3 trillion routes. The Brute Force approach does not use heuristics or caching; it literally adds up the distances for all 1.3 trillion permutations to find the smallest number, which will crash modern computers.

Use Case Breakdown: Recursive Subset Sum
Time: $O(2^n)$ Space: $O(n)$

What it says: Given an array of integers, is there any combination of numbers that exactly matches a target sum? The Brute Force approach uses recursion to explore an "Include" branch and an "Exclude" branch for every single number.

Why it is Brute Force: It creates a massive binary recursion tree. It does not cache overlapping answers (which DP would do). Because the tree doubles in width at every level, the runtime explodes exponentially to $O(2^n)$.

Java: Recursive Subset Sum
public boolean isSubsetSum(int[] set, int n, int sum) {
    if (sum == 0) return true;
    if (n == 0) return false;

    // Last element is greater than sum, ignore it
    if (set[n - 1] > sum)
        return isSubsetSum(set, n - 1, sum);

    // Check inclusion branch OR exclusion branch
    return isSubsetSum(set, n - 1, sum) || 
           isSubsetSum(set, n - 1, sum - set[n - 1]);
}

Greedy Paradigm

The Strategy: The Greedy paradigm focuses on making the locally optimal choice at each stage with the hope of finding a global optimum. It acts with absolute confidence, "taking the biggest bite possible" and never looking back to reconsider its choices. Because it avoids complex branching, it is blazingly fast—often scaling at $O(n \log n)$ due to the necessity of sorting the data first. However, it will fail catastrophically and yield the wrong answer if the specific problem lacks the mathematical "Greedy Choice Property."
Use Case Breakdown: Standard Coin Change
Time: $O(n \log n)$ Space: $O(1)$

What it says: Make exact change using the minimum number of coins. Sort the coins from largest to smallest, and repeatedly subtract the largest coin that doesn't exceed the remaining balance.

Why it is Greedy: It makes the locally optimal choice (e.g., "grab a ₹50 note because it covers the most ground right now") without evaluating combinations. This works flawlessly for standard currencies, but fails if arbitrary coins (like a ₹15 coin) are introduced, trapping the algorithm in a bad local path.

Java: Greedy Coin Change
public void greedyChange(int[] coins, int amount) {
    // Assumption: coins array is sorted descending
    for (int coin : coins) {
        while (amount >= coin) {
            amount -= coin;
            System.out.print(coin + " ");
        }
    }
}

Use Case Breakdown: Fractional Knapsack
Time: $O(n \log n)$ Space: $O(1)$

What it says: A thief wants to maximize the value in their bag, but they are allowed to break items into fractions (like gold dust). The algorithm calculates the value-to-weight ratio for each item, sorts them, and greedily shovels the highest-ratio items into the bag.

Why it is Greedy: It relies entirely on the local ratio. The *only* reason Greedy succeeds here is because the items are divisible. If the items were solid objects (0/1 Knapsack), the greedy choice would leave empty, wasted space in the bag, failing the global optimum.

Java: Fractional Knapsack
public double getMaxValue(Item[] arr, int capacity) {
    // Sort arr by (value/weight) descending
    double totalValue = 0;
    for (Item i : arr) {
        if (capacity - i.weight >= 0) {
            capacity -= i.weight;
            totalValue += i.value;
        } else {
            double fraction = (double)capacity / i.weight;
            totalValue += (i.value * fraction);
            break; // Full
        }
    }
    return totalValue;
}

Use Case Breakdown: Activity Selection
Time: $O(n \log n)$ Space: $O(1)$

What it says: You have a single conference room and multiple meetings with different start and end times. The goal is to fit the maximum number of meetings into the day. The algorithm sorts the meetings by **finish time**, and always picks the one that ends the earliest.

Why it is Greedy: It doesn't look at the duration of the meetings or how they interlock. It purely makes the greedy assumption that finishing a meeting as early as possible guarantees the maximum amount of free time for subsequent meetings.

Java: Activity Selection
// Arrays start/finish sorted by finish time
public void printMaxActivities(int[] s, int[] f, int n) {
    int i = 0;
    System.out.print(i + " "); 
    for (int j = 1; j < n; j++) {
        if (s[j] >= f[i]) {
            System.out.print(j + " ");
            i = j; 
        }
    }
}

Use Case Breakdown: Kruskal's MST
Time: $O(E \log E)$ Space: $O(V)$

What it says: Build a Minimum Spanning Tree (MST) to connect a set of cities with the cheapest network of cables. Sort all the cables (edges) from cheapest to most expensive, and greedily pick the cheapest ones as long as they don't form a closed loop.

Why it is Greedy: At no point does Kruskal's evaluate the graph as a whole. It just looks at the sorted list of edges and takes the absolute cheapest local connection available. Due to the properties of spanning trees, this greedy local choice naturally builds the global minimum network.

Dynamic Programming (DP)

The Strategy: Dynamic Programming (DP) is essentially "Smart Brute-Force with a Memory". It is designed to solve complex problems by breaking them down into simpler, overlapping subproblems. By storing the results of these subproblems in an array or table (a technique called Memoization or Tabulation), DP completely avoids redundant calculations. This allows it to find mathematically perfect global optimums while dropping exponential time complexities down to linear or polynomial bounds.
Use Case Breakdown: Strange Coin Change
Time: $O(n \times \text{target})$ Space: $O(\text{target})$

What it says: Fix the blind spot of the Greedy algorithm. Given arbitrary coins like `[1, 4, 5]`, find the minimum coins for a target by building a 1D array from 0 up to the target, checking the absolute minimum combinations for every step.

Why it is DP: It relies on Optimal Substructure. To find the best way to make ₹8, it asks the array: "What is the best way to make ₹4?" It uses previously cached sub-answers to guarantee it doesn't get trapped by an early bad choice.

Java: DP Bottom-Up
public int dpCoinChange(int[] coins, int target) {
    int[] dp = new int[target + 1];
    Arrays.fill(dp, target + 1);
    dp[0] = 0; 
    for (int i = 1; i <= target; i++) {
        for (int c : coins) {
            if (i >= c) dp[i] = Math.min(dp[i], 1 + dp[i - c]);
        }
    }
    return dp[target];
}

Use Case Breakdown: Climbing Stairs
Time: $O(N)$ Space: $O(N)$

What it says: You can climb a staircase 1 or 2 steps at a time. To find the number of ways to reach step $N$, you simply add the number of ways you reached step $N-1$ and step $N-2$.

Why it is DP: Mathematically, this is the Fibonacci sequence. A naive recursive brute-force approach would branch exponentially ($O(2^n)$), recalculating the same steps over and over. DP builds a linear 1D array to store the result of each step, completely flattening the recursion into an $O(N)$ loop.

Java: Climbing Stairs DP
public int climbStairs(int n) {
    if (n == 1) return 1;
    int[] dp = new int[n + 1];
    dp[1] = 1; dp[2] = 2; 

    for (int i = 3; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }
    return dp[n];
}

Use Case Breakdown: 0/1 Knapsack
Time: $O(n \times W)$ Space: $O(n \times W)$

What it says: Maximize the value of stolen items within a weight capacity. Unlike the Fractional Knapsack, items cannot be broken. It builds a 2D matrix evaluating every item against every possible incremental weight limit.

Why it is DP: A Greedy approach fails because taking a high-ratio item might leave an awkward gap in the bag. The 2D DP table mathematically evaluates the overlapping subproblems: "Is my value higher if I exclude this item, or if I include it and add its value to the best combination that fits in the remaining space?"

Java: 0/1 Knapsack (2D DP)
public int knapSack(int W, int[] wt, int[] val, int n) {
    int[][] dp = new int[n + 1][W + 1];
    for (int i = 1; i <= n; i++) {
        for (int w = 1; w <= W; w++) {
            if (wt[i - 1] <= w) {
                dp[i][w] = Math.max(val[i - 1] + 
                           dp[i - 1][w - wt[i - 1]], dp[i - 1][w]);
            } else {
                dp[i][w] = dp[i - 1][w];
            }
        }
    }
    return dp[n][W];
}

Use Case Breakdown: Longest Common Subsequence
Time: $O(n \times m)$ Space: $O(n \times m)$

What it says: Find the longest matching sequence of characters between two strings. It builds a 2D matrix where the columns represent String A and the rows represent String B.

Why it is DP: Brute force would generate every possible subsequence ($O(2^n)$) and check them. DP avoids this by building on optimal substructures: If two characters match, it adds 1 to the diagonal cell. If they don't match, it carries forward the maximum sequence length found so far from either the left or top cell.

Java: LCS (2D DP)
public int lcs(String s1, String s2) {
    int m = s1.length(), n = s2.length();
    int[][] dp = new int[m + 1][n + 1];

    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (s1.charAt(i - 1) == s2.charAt(j - 1))
                dp[i][j] = dp[i - 1][j - 1] + 1; // Diagonal
            else
                dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
        }
    }
    return dp[m][n];
}
