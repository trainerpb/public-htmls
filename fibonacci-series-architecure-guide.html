The Recursive Trap: $O(2^n)$

The Fibonacci sequence is defined elegantly: a number is the sum of the two preceding ones. Translating this mathematical definition directly into recursive code creates a massive performance trap.

The Problem: Overlapping Subproblems
Because the function branches out, it recalculates the exact same branches thousands of times. Calculating `fib(50)` recursively would require roughly 1.25 billion function calls. 
public long fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);
}
long long fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);
}
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)

Iterative (Bottom-Up): $O(n)$

Instead of starting at the top and branching down, we start at the base cases (0 and 1) and build our way up using a standard loop. This completely eliminates the call stack and drops the time complexity from exponential to linear.

Space Optimization: We only ever need the last two numbers to calculate the next one. We don't need an array. We can accomplish this with $O(1)$ constant space using variables. 
public long fib(int n) {
    if (n <= 1) return n;
    long a = 0, b = 1, sum = 0;
    for (int i = 2; i <= n; i++) {
        sum = a + b;
        a = b;
        b = sum;
    }
    return b;
}
long long fib(int n) {
    if (n <= 1) return n;
    long long a = 0, b = 1, sum = 0;
    for (int i = 2; i <= n; i++) {
        sum = a + b;
        a = b;
        b = sum;
    }
    return b;
}
def fib(n):
    if n <= 1:
        return n
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b

Memoization (Top-Down Caching): $O(n)$

If we want to keep the elegant, readable recursive structure, we can fix the performance issue by adding a Cache (usually an array or Hash Map). Before calculating `fib(n)`, we check if we already have the answer. If we do, we return it instantly.

Dynamic Programming: This technique prevents the tree from branching endlessly. The space complexity increases to $O(n)$ to store the cache, but the time complexity drops to $O(n)$. 
public long fib(int n, long[] memo) {
    if (n <= 1) return n;
    
    // Check Cache
    if (memo[n] != 0) return memo[n]; 
    
    // Calculate and Store
    memo[n] = fib(n - 1, memo) + fib(n - 2, memo);
    return memo[n];
}
long long fib(int n, long long* memo) {
    if (n <= 1) return n;
    if (memo[n] != 0) return memo[n];
    
    memo[n] = fib(n - 1, memo) + fib(n - 2, memo);
    return memo[n];
}
def fib(n, memo={}):
    if n <= 1:
        return n
    if n not in memo:
        memo[n] = fib(n - 1, memo) + fib(n - 2, memo)
    return memo[n]

Compare & Contrast

Approach Time / Space Recurrence Relation & Notes
1. Naive Recursive Time: $O(2^n)$
Space: $O(n)$		 Formula: $$T(n) = T(n-1) + T(n-2) + O(1)$$

Fails the Master Theorem because of subtraction. True tight bound relies on the Golden Ratio: $O(1.618^n)$. Catastrophic for production.
2. Memoization (Top-Down) Time: $O(n)$
Space: $O(n)$		 Formula: $$T(n) = O(1) \text{ (if cached)}$$

Excellent for complex DP problems where state spaces are sparse. Requires memory allocation for the cache array and maintains stack overhead.
3. Iterative (Bottom-Up) Time: $O(n)$
Space: $O(1)$		 Formula: $$\sum_{i=2}^{n} O(1) = O(n)$$

The production standard. Zero risk of `StackOverflowError`. Minimal memory footprint via variable swapping.

Terms That Frighten!

Computer Science academia loves using intimidating words for concepts that are actually quite simple and logical. Here is a translation of what these scary words actually mean in plain English.

Brute Force Hin: पाशविक बल / सीधा तरीका
Ben: পাশবিক শক্তি / সরল পদ্ধতি

What it sounds like: Smashing a computer with a hammer.

What it actually is: The "try everything" approach. It's the most naive, straightforward way to solve a problem without any clever optimizations.

Example: Trying every possible 4-digit combination to unlock a padlock until one works.

Recursion Hin: पुनरावृत्ति (Punravrutti)
Ben: পুনরাবৃত্তি (Punarabritti)

What it sounds like: A complex mathematical loop paradox.

What it actually is: A function that simply calls itself to solve a smaller piece of the same problem until it reaches an obvious stopping point (the base case).

Example: Looking up a word in a dictionary, which tells you to look up another word, until you finally find one you understand.

Divide and Conquer Hin: फूट डालो और राज करो
Ben: ভাগ করো ও শাসন করো

What it sounds like: An ancient military strategy.

What it actually is: Taking a massive problem, slicing it in half, slicing those halves in half, solving the tiny pieces, and then gluing the answers back together.

Example: Merge Sort. Instead of sorting 1,000 papers, you give 500 to a friend, who gives 250 to another friend, until everyone is just sorting 2 papers.

Dynamic Programming Hin: गतिशील प्रोग्रामिंग
Ben: গতিশীল প্রোগ্রামিং

What it sounds like: Writing code while running on a treadmill.

What it actually is: "Smart Remembering". It's just solving a problem but writing down the answer to intermediate steps on a notepad so you don't have to calculate them again later (Memoization).

Example: If I ask you what 10 + 10 is, you say 20. If I immediately ask what 10 + 10 + 1 is, you don't start from zero; you just remember 20 and add 1.

Greedy Algorithm Hin: लालची एल्गोरिदम
Ben: লোভী অ্যালগরিদম

What it sounds like: Code written by a corporate billionaire.

What it actually is: An algorithm that makes the absolute best short-term choice at every single step, hoping those short-term wins add up to the best overall long-term solution (it doesn't look ahead).

Example: Giving someone $0.90 in change. A greedy algorithm grabs the biggest coin first (a 50-cent piece), then a 25-cent piece, then a 10, then a 5. It is "greedy" for the biggest immediate chunk.
