Properly Understanding the 3 Methods

To analyze a recursive algorithm, we express its time complexity as a mathematical sequence called a recurrence relation. We have three distinct tools to solve these equations.

1. The Substitution Method (Mathematical Proof)

This is a formal proof by mathematical induction. It does not give you the answer directly; you must guess the answer first, and then use algebra to prove your guess is strictly bounded by a constant.

  1. Guess: Formulate an intuition-based guess for the Big O bound.
  2. Assume: Assume your guess is true for a smaller sub-problem (e.g., $n-1$ or $n/2$).
  3. Substitute: Plug that assumption back into the original equation.
  4. Prove: Simplify the algebra to prove that $T(n) \le c \cdot f(n)$, isolating the constant $c$ to show the bound holds.

2. The Recursion Tree Method (Visual Summation)

Instead of algebra, we draw the execution of the algorithm as a branching tree. This is highly effective when substitution algebra gets messy.

  • Draw the Nodes: Map out how the problem splits.
  • Sum the Levels: Calculate the total non-recursive work done horizontally across each level.
  • Find the Height: Determine how deep the tree goes before hitting the base case (usually $\log n$ or $n$).
  • Calculate: $\text{Total Work} = \text{Sum of all levels}$.

3. The Master Theorem (The Cookbook Formula)

The fastest method. It provides an immediate formula for algorithms that split into fractions. It compares the work done at the root vs. the work done at the leaves.

For $T(n) = aT(n/b) + f(n)$, we calculate the "Leaf Work" using $n^{\log_b a}$:

  • Case 1: Leaves are heavier $\implies \Theta(n^{\log_b a})$
  • Case 2: Leaves and Root are perfectly tied $\implies \Theta(n^{\log_b a} \log n)$
  • Case 3: Root is heavier $\implies \Theta(f(n))$

Solving for $c$: $T(n) = T(n-1) + n$

Let's use the Substitution Method to rigorously solve a classic sum-of-integers recurrence relation. The most common point of confusion is proving the constant $c$. Let's break it down step-by-step.

The Equation: $$T(n) = T(n-1) + n$$ The Guess: $O(n^2)$
The Goal: Prove mathematically that $T(n) \le c \cdot n^2$ for some positive constant $c$.

Step 1: The Induction Assumption

Assume our guess holds true for the smaller input, $n-1$.

$$T(n-1) \le c(n-1)^2$$

Step 2: Substitution & Expansion

Plug our assumption into the original equation:

$$T(n) \le c(n-1)^2 + n$$

Expand the quadratic $(n-1)^2$:

$$T(n) \le c(n^2 - 2n + 1) + n$$

Distribute the $c$:

$$T(n) \le cn^2 - 2cn + c + n$$

Step 3: Isolating the Target

We want to prove the whole equation is $\le cn^2$. Let's pull $cn^2$ to the front, and group the leftover "baggage" together:

$$T(n) \le cn^2 - n(2c - 1) + c$$

Step 4: Finding the Value of $c$

For our proof to succeed, the "baggage" must not add extra weight. It must be less than or equal to zero:

$$- n(2c - 1) + c \le 0$$

Let's logically test a value for $c$. What happens if we pick $c = 1$?

$$- n(2(1) - 1) + 1 \le 0$$ $$- n(1) + 1 \le 0$$ $$- n + 1 \le 0$$

Move $n$ to the other side:

$$1 \le n$$

✔ The Proof is Complete

In algorithm analysis, input size $n$ is always a positive integer ($n \ge 1$). Because $1 \le n$ is inherently a true statement, our choice of $c = 1$ works perfectly.

We have rigorously proven that $T(n) \le 1 \cdot n^2$. The exact time complexity is mathematically verified as $O(n^2)$.

The Master Theorem Doesn't Always Work!

Students frequently try to force every single equation into the Master Theorem. It is a powerful tool, but it has strict limitations. It will mathematically crash if you ignore these rules.

⚠ When the Master Theorem FAILS:

  1. Subtraction in the Argument: If the equation uses $T(n-1)$ or $T(n-2)$, the theorem breaks instantly. The input size $b$ must be a division factor strictly greater than 1 ($b > 1$).
  2. Non-Polynomial $f(n)$: If the extra work outside the recursive call is exponential (e.g., $2^n$) or a trigonometric function, it cannot be reliably compared to $n^{\log_b a}$.
  3. Asymptotic Gaps: The difference between $f(n)$ and the critical bound must be polynomial ($n^\epsilon$). A logarithmic difference (like comparing $n$ against $n \log n$) falls into a gap where the basic theorem cannot be evaluated.

What to do when it fails? Immediately switch to the Recursion Tree Method or the Substitution Method.

Divide vs. Reduce and Conquer

Recognizing the difference between these two paradigms dictates whether you are allowed to use the Master Theorem, and dictates the fundamental shape of the execution tree.

Feature Divide and Conquer Reduce (Decrease) & Conquer
The Core Logic Splits the problem into fractions. Subtracts a constant amount from the problem.
The Recurrence Form $T(n) = aT(n/b) + f(n)$ $T(n) = aT(n-c) + f(n)$
Data Shrinks By... Division (e.g., $n/2$) Subtraction (e.g., $n-1$)
Tree Depth (Height) Logarithmic: $\log_b n$ Linear: $n / c$
Master Theorem? YES. Perfectly applicable. NO. Master Theorem is invalid.
Algorithm Example Merge Sort $O(n \log n)$ Recursive Fibonacci $O(2^n)$
Rule of Thumb: If you see a division symbol ($/$), reach for the Master Theorem. If you see a minus sign ($-$), draw the Recursion Tree!
