Analysis of Recursive Algorithms

When an algorithm calls itself to solve smaller sub-problems, we cannot analyze its time complexity using simple loops. Instead, we use a Recurrence Relation: an equation that recursively defines a sequence.

Chalk

Consider Merge Sort. It splits an array of size $n$ into two halves, sorts them recursively, and then merges them. We express its time mathematically as: $$T(n) = 2T(n/2) + O(n)$$

Talk

This means: "The time to solve a problem of size $n$ equals $2$ times the time to solve a problem of size $n/2$, plus $O(n)$ time to merge the halves together."

Do

To find the Big O bound, we must solve this recurrence relation. We have three primary weapons in our mathematical arsenal:

  • Substitution Method: Guessing the bound and proving it.
  • Recursion Tree Method: Drawing the execution branches.
  • Master's Theorem: The ultimate formula shortcut.

Verify

We verify our findings by ensuring the calculated asymptotic bound correctly matches real-world execution times as $n \to \infty$.

Solving: The Substitution Method

The substitution method relies on mathematical induction. It is a "guess and check" strategy.

Chalk

Let's solve the recurrence: $$T(n) = T(n-1) + n$$

Talk

We will guess that the solution is $O(n^2)$. To prove this, we must show that $T(n) \le c \cdot n^2$ for an appropriate constant $c > 0$.

Do

Assume our guess holds true for smaller values, specifically for $n-1$: $$T(n-1) \le c(n-1)^2$$

Substitute this assumption back into the original equation:

$$T(n) \le c(n-1)^2 + n$$ $$T(n) \le c(n^2 - 2n + 1) + n$$ $$T(n) \le cn^2 - 2cn + c + n$$ $$T(n) \le cn^2 - n(2c - 1) + c$$

Verify

For $T(n) \le cn^2$ to be true, the remaining term $-n(2c - 1) + c$ must be less than or equal to $0$. As long as we choose a constant $c \ge 1$, this condition holds true. Therefore, our guess is verified: $T(n) = O(n^2)$.

Solving: Recursion Tree Method

The recursion tree visualizes what happens as recursive calls unfold. We calculate the cost of work done at each level of the tree, and then sum the levels together.

Chalk

Let's solve Merge Sort's recurrence: $$T(n) = 2T(n/2) + n$$

Talk

The root of the tree represents a cost of $n$. It splits into two nodes, each costing $n/2$. Those split into four nodes, each costing $n/4$.

Do

  • Level 0 Cost: $n$
  • Level 1 Cost: $(n/2) + (n/2) = n$
  • Level 2 Cost: $(n/4) + (n/4) + (n/4) + (n/4) = n$

Notice a pattern? The cost at every level is exactly $n$.

Now, how deep is the tree? Since we divide $n$ by 2 at each level, the tree hits the base case ($size = 1$) at a depth of $\log_2 n$.

Verify

Total Cost = (Cost per level) $\times$ (Number of levels).

$$Total = n \times \log_2 n = O(n \log n)$$

Solving: Master's Theorem

The Master Theorem is a direct formula providing a "cookbook" solution for recurrence relations of the form:

$$T(n) = aT(n/b) + f(n)$$

Where:
• $a \ge 1$: Number of sub-problems.
• $b > 1$: Factor by which input size shrinks.
• $f(n)$: Cost of the work done outside the recursive calls (dividing/merging).

Chalk & Talk: The Three Cases

We determine the result by comparing $f(n)$ against a critical polynomial bound: $n^{\log_b a}$.

Case Condition Result $T(n)$ Real-Life Scenario
1. Leaves Dominate $f(n)$ is smaller than $n^{\log_b a}$ $\Theta(n^{\log_b a})$ Tree Traversals where splitting is cheap.
2. Even Balance $f(n)$ is exactly equal to $n^{\log_b a}$ $\Theta(n^{\log_b a} \log n)$ Merge Sort. Work is distributed evenly.
3. Root Dominates $f(n)$ is larger than $n^{\log_b a}$ $\Theta(f(n))$ Inefficient combining algorithms.

Do & Verify

Solve: $T(n) = 8T(n/2) + 1000n^2$

  • $a = 8, b = 2, f(n) = 1000n^2$
  • Critical bound: $n^{\log_2 8} = n^3$
  • Compare: $f(n)$ is $O(n^2)$, which is smaller than $n^3$.
  • This is Case 1. The bound is $O(n^3)$.

Visualizing Recurrence Bounds

Understanding what these solved relations mean is crucial. A simple tweak in a recursive algorithm can shift its execution from $O(n)$ to $O(n^2)$.
