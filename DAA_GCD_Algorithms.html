Greatest Common Divisor (GCD)

Finding the Greatest Common Divisor is a classic algorithmic problem. While the mathematical concept remains the same, the implementation approach drastically changes the performance, especially when handling massive enterprise-scale integers.

Visualizing the Euclidean Algorithm

Before looking at code, we must understand the math visually. The Euclidean approach essentially works by tiling a rectangle of dimensions $a \times b$ with the largest possible square, repeatedly dealing with the remainder until the area is perfectly filled.

The Core Theorem: The GCD of two numbers also divides their difference. Therefore, calculating $GCD(a, b)$ is mathematically identical to calculating $GCD(b, a \bmod b)$.

The Naive Approaches

1. Brute Force

Iterate backward from the smaller number until you find a number that divides both evenly. 

int gcd(int a, int b) {
    int limit = Math.min(a, b);
    for (int i = limit; i >= 1; i--) {
        if (a % i == 0 && b % i == 0)
            return i;
    }
    return 1;
}
int gcd(int a, int b) {
    int limit = (a < b) ? a : b;
    for (int i = limit; i >= 1; i--) {
        if (a % i == 0 && b % i == 0)
            return i;
    }
    return 1;
}
def gcd(a, b):
    limit = min(a, b)
    for i in range(limit, 0, -1):
        if a % i == 0 and b % i == 0:
            return i
    return 1

Time: $O(\min(a, b))$ | Space: $O(1)$

2. Euclid (Subtraction)

Repeatedly subtract the smaller number from the larger one until they are equal.

int gcd(int a, int b) {
    if (a == 0) return b;
    if (b == 0) return a;
    while (a != b) {
        if (a > b) a -= b;
        else b -= a;
    }
    return a;
}
def gcd(a, b):
    if a == 0: return b
    if b == 0: return a
    while a != b:
        if a > b:
            a -= b
        else:
            b -= a
    return a

Time: $O(\max(a, b))$ | Space: $O(1)$

The Optimal Approaches

3. Euclid (Modulo Recursive)

Instead of subtracting, use the modulo operator to jump straight to the remainder.

int gcd(int a, int b) {
    return b == 0 ? a : gcd(b, a % b);
}
def gcd(a, b):
    return a if b == 0 else gcd(b, a % b)

Time: $O(\log(\min(a, b)))$ | Space: $O(\log(\min(a, b)))$

Note: Space complexity comes from the call stack depth.

4. Euclid (Modulo Iterative)

The recursive logic unwound into a highly optimized standard `while` loop.

int gcd(int a, int b) {
    while (b != 0) {
        int temp = b;
        b = a % b;
        a = temp;
    }
    return a;
}
def gcd(a, b):
    while b != 0:
        a, b = b, a % b  # Tuple unpacking!
    return a

Time: $O(\log(\min(a, b)))$ | Space: $O(1)$

The Architect's Comparison Table

If you are writing clean architecture or a backend utility class, Method 4 (Modulo Iterative) is the undisputed winner. It balances memory safety, execution speed, and readability without risking thread stack depths.

Approach Complexity Pros & Cons
1. Brute Force Time: $O(\min(a, b))$
Space: $O(1)$		 Pros: Extremely easy to read and understand. Requires no math knowledge.

Cons: Terrible performance. `gcd(1000000000, 999999999)` runs a billion loops.
2. Euclid (Subtraction) Time: $O(\max(a, b))$
Space: $O(1)$		 Pros: Avoids modulo/division operations, historically expensive on old CPUs.

Cons: Worst-case is catastrophic. `gcd(1000000, 1)` takes 1 million iterations.
3. Euclid (Modulo Recursive) Time: $O(\log(\min(a, b)))$
Space: $O(\log(\min(a, b)))$		 Pros: Beautiful, mathematical, and concise 1-liner.

Cons: Uses the call stack. Method invocation carries overhead compared to raw loops.
4. Euclid (Modulo Iterative) Time: $O(\log(\min(a, b)))$
Space: $O(1)$		 Pros: The Enterprise Standard. Massive logarithmic speedup without memory overhead.

Cons: Slightly more verbose in languages that don't support tuple unpacking.
